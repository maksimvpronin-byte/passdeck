# PROJECT PROMPT — PassDeck

## Текущий статус реализации

Версия разработки: `0.2.0`, ветка `dev/0.2.0`.

Реализовано на текущем этапе:

- рабочий локальный KDBX-каркас 0.1.x;
- Windows Auto-Type по `Ctrl+Alt+A`;
- пользовательские поля KDBX;
- защищённые пользовательские поля с явным показом и копированием;
- сохранение скрытых значений без передачи в renderer при обычном редактировании;
- вложения KDBX;
- перенос записей между группами через drag-and-drop.

Этап 0.2.0 функционально завершён. История записей, корзина, перенос групп и Save As не входят в продуктовый интерфейс PassDeck. Новые функции добавляются только после отдельного согласования.

## 1. Роль исполнителя

Ты — ведущий разработчик desktop-приложения и security engineer. Продолжай разработку PassDeck последовательно, небольшими проверяемыми итерациями. Нельзя заменять согласованные требования упрощёнными аналогами без явной технической причины. Безопасность данных и обратная совместимость с KeePass важнее скорости реализации.

Перед каждым изменением:

1. изучи существующую архитектуру и тесты;
2. определи минимальный безопасный объём патча;
3. не ломай уже работающие KDBX-функции;
4. добавь или обнови тесты;
5. выполни `lint`, `typecheck`, `test`, `build`;
6. не добавляй сетевые запросы, телеметрию или файловое логирование.

## 2. Назначение продукта

PassDeck — полностью локальный менеджер паролей, аналогичный KeePass по назначению, но с современным интерфейсом в стиле KubeDeck.

Ключевые ограничения:

- локальное хранение без аккаунтов и собственного сервера;
- основной формат данных — стандартный файл `.kdbx`;
- пользователь может открывать существующие базы KeePass и переносить их между приложениями;
- Windows выпускается только в portable-варианте;
- macOS добавляется после стабилизации Windows-версии;
- Linux пока не поддерживается;
- название приложения: `PassDeck`;
- репозиторий: `passdeck`;
- App ID и Bundle ID: `com.maksimpronin.passdeck`;
- версия первого каркаса: `0.1.0`;
- лицензия: MIT.

## 3. Поддерживаемые платформы

### Windows

- Windows 10 x64 и Windows 11 x64;
- единственный артефакт: `PassDeck-Portable-<version>-x64.exe`;
- без установщика;

- без автоматической ассоциации `.kdbx`;
- настройки portable-версии находятся рядом с EXE в `data/`.

### macOS

- первым реализовать Apple Silicon `arm64`;
- затем Intel `x64`;

- Touch ID реализовать на macOS-этапе;
- секрет для Touch ID хранить только через macOS Keychain;
- при любой ошибке Touch ID требовать мастер-пароль;
- первая сборка может быть без подписи и notarization, но конфигурация должна позволять добавить их позже.

## 5. Архитектурные границы безопасности

Обязательные параметры Electron:

- `contextIsolation: true`;
- `nodeIntegration: false`;
- `sandbox: true`;
- строгий CSP;
- запрет навигации renderer;
- запрет загрузки удалённого контента;
- строго типизированный preload API;
- IPC только по белому списку;
- проверка и нормализация всех путей и входных данных в main;
- криптография и файловая система доступны только main-процессу.

Renderer не должен получать расшифрованную базу целиком. Он получает только данные, необходимые для текущего экрана. Пароль записи передаётся отдельно только при явном показе, копировании или Auto-Type.

Никогда не записывать в логи или настройки:

- мастер-пароли;
- пароли записей;
- ключи шифрования;
- TOTP-секреты;
- содержимое защищённых пользовательских полей;
- содержимое заметок;
- KDBX payload.

Файловый журнал событий не создавать вообще. Ошибки показывать только в UI; технические детали можно вручную скопировать из диалога.

## 6. KDBX и криптография

Не реализовывать KDBX или криптографические алгоритмы самостоятельно. Использовать зрелую библиотеку, проверяя лицензию, активность, тесты и поддержку нужных функций.

Требования:

- открытие и сохранение KDBX 3.1 и KDBX 4.x;
- новые базы создаются как KDBX 4.1;
- шифрование новых баз: AES-256;
- KDF новых баз: Argon2id;
- параметры Argon2id должны подбираться под производительность компьютера на отдельном этапе;
- при сохранении существующей базы максимально сохранять её исходные параметры и неизвестные метаданные;
- поддерживать группы, записи, теги, заметки, пользовательские поля, вложения и пользовательские иконки;
- сохранять совместимость с внутренними метаданными KDBX, но не добавлять UI истории записей или корзины;
- способы разблокировки key file и аппаратные ключи в первой версии не поддерживаются;
- при неподдерживаемом способе разблокировки показывать отдельную понятную ошибку;
- исходный файл никогда не изменять при ошибке открытия;
- не пытаться автоматически чинить и перезаписывать повреждённую базу.

Для нового мастер-пароля:

- минимум 8 символов;
- пустой пароль запрещён;
- показать индикатор стойкости;
- предупредить о слабом пароле;
- не навязывать обязательные цифры, регистр или спецсимволы;
- поддерживать длинные парольные фразы;
- требовать повторный ввод;
- явно предупреждать, что восстановить забытый мастер-пароль невозможно;
- не делать подсказку, контрольные вопросы, recovery key или сброс пароля.

Вставку мастер-пароля из буфера разрешить. PassDeck не копирует мастер-пароль в буфер и не очищает чужое содержимое буфера после разблокировки. Поле мастер-пароля очищается сразу после использования.

После пяти подряд неверных попыток вводить задержку 5 секунд. Количество попыток не ограничивать. Историю попыток не хранить.

## 7. Файловые операции и сохранение

Основной режим:

- нет автосохранения после каждого изменения;
- пользователь может сохранить вручную;
- перед нормальным закрытием изменённые базы автоматически сохраняются;
- при закрытии нескольких вкладок сохранить все изменённые базы;
- если хотя бы одна база не сохранилась, отменить закрытие всего приложения;
- показать список проблемных баз и предложить повторить сохранение или оставить базы открытыми без потери данных;
- при ошибке данные остаются в памяти.

Сохранение должно быть атомарным:

1. проверить внешнее изменение исходного файла;
2. создать зашифрованную резервную копию существующего `.kdbx`;
3. записать новый файл во временный файл в той же директории;
4. выполнить flush/fsync;
5. атомарно заменить оригинал;
6. обновить fingerprint.

Резервные копии:

- создавать перед каждым сохранением существующего файла;
- хранить только две последние копии на базу;
- каталог portable-версии: `data/backups`;
- имя содержит название базы, дату и время;
- копии остаются полноценными зашифрованными `.kdbx` с тем же мастер-паролем;
- добавить UI просмотра и восстановления.

Recovery-копии:

- функция настраиваемая и выключена по умолчанию;
- при включении первая зашифрованная recovery-копия создаётся через 60 секунд после первого несохранённого изменения;
- далее обновляется не чаще одного раза в 60 секунд;
- хранить одну recovery-копию на базу в `data/recovery`;
- основной файл не изменять;
- после успешного сохранения удалить recovery;
- при запуске предложить восстановить более свежую recovery-копию.

## 8. Блокировки и конфликт файлов

- приложение работает в одном экземпляре;
- повторный запуск активирует существующее окно;
- переданные `.kdbx` открываются вкладками в существующем процессе;
- рядом с открытой базой создаётся временный `.passdeck.lock`;
- если lock существует, открыть базу только для чтения;
- показать информацию о блокировке;
- разрешить принудительное открытие после явного предупреждения;
- перед сохранением сравнивать размер и mtime/hash исходного файла;
- при внешнем изменении не перезаписывать файл и показать понятную ошибку; Save As не реализовывать.

## 9. Модель данных записи

Запись поддерживает:

- название;
- логин;
- пароль;
- URL;
- обычный текст заметки без Markdown;
- пользовательские поля, включая защищённые;
- теги;
- признак избранного;
- срок действия пароля;
- вложения;
- стандартную и пользовательскую иконку;
- фиксированную последовательность Auto-Type.

Внутренняя история KDBX может сохраняться библиотекой для совместимости, но отдельный пользовательский интерфейс просмотра или восстановления версий не реализуется.

Вложения:

- хранятся внутри `.kdbx`;
- максимум 25 МБ на один файл;
- максимум 100 МБ суммарно на запись;
- интерфейс получает только имя и размер, байты остаются в main-процессе;
- экспортировать на диск только по явной команде через системный диалог;
- не заменять вложение с совпадающим именем без явного удаления существующего файла.

Дублирование записи:

- команда «Создать копию»;
- добавить «Копия» к названию;
- скопировать поля, вложения и иконку;
- не копировать историю;
- создать новый UUID.

Undo/Redo в первой версии не делать.

## 10. Группы и списки

- древовидные группы и подгруппы;
- перенос записей выполняется только drag-and-drop на другую группу в дереве;
- кнопки и модальные окна перемещения не добавлять; перенос групп не реализовывать;
- удаление записи выполняется напрямую только после подтверждения;
- корзина и восстановление удалённых объектов в интерфейсе не реализуются.

Быстрые фильтры:

- все записи;
- избранные;
- просроченные;
- слабые пароли;
- повторяющиеся пароли;
- недавно изменённые;
- с вложениями;
- по тегам.

Таблица записей по умолчанию:

- название;
- логин;
- URL;
- группа;
- теги;
- срок действия;
- дата последнего изменения;
- избранное.

Разрешить скрывать столбцы, менять порядок и сортировку.

## 11. Поиск

Два режима:

- по текущей базе;
- глобально по всем открытым и разблокированным базам.

Поиск включает:

- название;
- логин;
- URL;
- заметки;
- теги;
- пользовательские поля;
- название группы.

Не искать по:

- паролям;
- вложениям.

Заблокированные базы не участвуют. Поисковые запросы не сохранять.

## 12. Несколько баз во вкладках

- несколько `.kdbx` одновременно во вкладках одного окна;
- каждая вкладка имеет отдельный мастер-пароль, lock-файл, dirty-state и состояние блокировки;
- независимое сохранение и проверка внешних изменений;
- отдельное закрытие вкладки;
- после перезапуска можно восстановить вкладки прошлого сеанса заблокированными;
- поведение запуска настраиваемое;
- пользователь может разблокировать текущую вкладку или открыть общий экран заблокированных баз;
- каждую базу разблокировать своим мастер-паролем.

## 14. Пароли, буфер и блокировка

Пароль записи скрыт по умолчанию. Кнопка с глазом показывает его максимум на 10 секунд. При смене записи, вкладки или блокировке скрывать сразу.

Копирование:

- одно нажатие без предварительного раскрытия;
- уведомление «Пароль скопирован» без значения;
- пароль очищать через 30 секунд;
- логин очищать через 60 секунд;
- таймеры настраиваемые и отключаемые;
- перед очисткой проверить, что в буфере всё ещё значение PassDeck;
- не удалять данные, скопированные позже другим приложением;
- повторное копирование перезапускает таймер.

Автоблокировка:

- по умолчанию через 10 минут бездействия;
- настраиваемая и отключаемая;
- перед блокировкой сохранить изменения;
- при ошибке сохранения оставить базу открытой и показать ошибку;
- ручная блокировка текущей базы и всех баз;
- блокировка при закрытии вкладки и приложения;
- системные события sleep/session lock в первой версии не использовать.

Режим приватности, блокировку скриншотов и маскировку окна в первой версии не делать.

## 15. Генератор паролей

Встроенный генератор поддерживает:

- длину;

- цифры;

- исключение похожих символов;

Полностью локально проверять:

- слабые пароли;

- просроченные записи;
- записи без логина или URL.

Не использовать онлайн API и не поддерживать локальную базу скомпрометированных паролей в первой версии.

## 18. Auto-Type и URL

Первая версия:

- открыть URL системным браузером;
- копировать логин/пароль;
- глобальная горячая клавиша;
- Auto-Type в активное окно;
- браузерное расширение отложить.

Последовательность по умолчанию:

```text
{USERNAME}{TAB}{PASSWORD}{ENTER}
```

Текущий UX Auto-Type:

- использовать фиксированную последовательность `{USERNAME}{TAB}{PASSWORD}{ENTER}`;
- не показывать настройки Auto-Type в редакторе записи;
- по глобальной горячей клавише сразу выполнить Auto-Type в текущем активном окне без подтверждения;
- расширенные последовательности и индивидуальное отключение отложить до отдельного этапа;
- на macOS запросить Accessibility.

URL:

- `http://` и `https://` открывать системным браузером;
- нестандартные протоколы (`ssh://`, `rdp://`, `file://`) требуют подтверждения;
- `javascript:` и аналогичные опасные схемы блокировать;
- никогда не загружать сайты внутри PassDeck.

## 19. Смена мастер-пароля

Внутренние KDBX-снимки могут сохраняться библиотекой для совместимости, но отдельный UI истории, сравнения и восстановления версий не реализуется.

Смена мастер-пароля:

- запросить текущий;
- дважды запросить новый;
- минимум 8 символов;
- показать оценку стойкости;
- создать резервную копию;
- полностью перешифровать базу;
- при ошибке оставить исходный файл без изменений;
- после успеха заблокировать вкладку и потребовать новый пароль.

## 20. Горячие клавиши

Windows/Linux-style:

- `Ctrl+N` — новая запись;
- `Ctrl+S` — сохранить;
- `Ctrl+F` — поиск;
- `Ctrl+L` — копировать логин;
- `Ctrl+P` — копировать пароль;
- `Ctrl+Shift+G` — генератор;
- `Ctrl+Shift+L` — блокировка;
- глобальная комбинация Auto-Type.

На macOS использовать эквиваленты с `Cmd`, кроме глобальных комбинаций, которые должны быть настраиваемыми.

## 21. Системный трей

Настраиваемый режим. По умолчанию закрытие завершает приложение. При включении трея:

- показать/скрыть окно;
- заблокировать все базы;
- открыть последнюю базу;
- завершить PassDeck;
- не показывать названия баз и записей в меню;
- восстановленные вкладки остаются заблокированными.

## 22. Производительность

Цели:

- до 20 000 записей в базе;
- технически до 100 вкладок;
- поиск по 10 000 записей менее секунды на типичном компьютере;
- UI не блокируется при KDF, открытии, сохранении и импорте;
- тяжёлые операции выполнять в worker thread/utility process;
- показывать прогресс для больших вложений;
- не показывать частично расшифрованные данные до успешной проверки целостности.

## 23. Тестирование

Обязательны:

- unit-тесты групп, записей, настроек и валидации;
- интеграционные тесты открытия и сохранения KDBX;
- fixtures KDBX 3.1, 4.0 и 4.1;
- проверка повторного открытия сохранённого файла;
- проверка открытия в KeePass/KeePassXC в отдельном compatibility suite;
- тесты вложений, переноса записей, резервных копий, recovery и конфликтов;
- тесты очистки буфера;
- тесты блокировки;
- тесты IPC и запрета лишних каналов;
- Playwright E2E основных пользовательских сценариев;
- только вымышленные тестовые данные;
- запрет сборки при ошибках TypeScript, lint или тестов.

## 24. Сборка и поставка

Канонические скрипты:

- `build.bat` — основной запуск Windows-сборки;
- `scripts/build-portable-windows.ps1` — единая логика Windows-сборки;
- `build-macos.sh`.

Windows-скрипт одной командой:

1. проверяет Node/npm;
2. использует существующие зависимости и выполняет `npm ci` только при явном запросе или их отсутствии;
3. запускает audit, lint, typecheck и tests;
4. очищает старую сборку;
5. собирает Electron;
6. создаёт `PassDeck-Portable-<version>-x64.exe`;
7. кладёт результат в `release`.

macOS-скрипт:

- запускается непосредственно на Mac;
- собирает ARM `.app`, `.dmg`, `.zip`;
- не пытается подписывать без сертификата;
- готов к последующему добавлению signing/notarization.

Сертификаты, пароли и signing secrets никогда не добавлять в репозиторий.

## 26. Definition of Done для каждого патча

Патч готов только если:

- требования реализованы без скрытого сетевого поведения;
- секреты не попадают в настройки, логи, исключения или тестовые snapshots;
- IPC минимален и типизирован;
- существует негативный тест;
- `npm audit --audit-level=high` проходит;
- `npm run lint` проходит;
- `npm run typecheck` проходит;
- `npm run test` проходит;
- `npm run build` проходит;
- документация обновлена;
- пользовательские данные не мигрируются или не перезаписываются без резервной копии.
