# Архитектура PassDeck

## Процессы

```text
┌──────────────────────────────────────────────────────────────┐
│ Renderer: React UI                                           │
│ - без Node.js                                                │
│ - без файловой системы                                      │
│ - без прямой криптографии                                    │
│ - только window.passdeck API                                 │
└──────────────────────────────┬───────────────────────────────┘
                               │ contextBridge + ipcRenderer
┌──────────────────────────────▼───────────────────────────────┐
│ Preload                                                      │
│ - фиксированный типизированный белый список IPC              │
│ - не экспортирует ipcRenderer целиком                        │
└──────────────────────────────┬───────────────────────────────┘
                               │ ipcMain.handle
┌──────────────────────────────▼───────────────────────────────┐
│ Electron Main                                                │
│ - DatabaseService                                            │
│ - SettingsStore                                              │
│ - KDBX/Argon2                                                │
│ - атомарная запись, backup, lock                             │
│ - clipboard и OS-интеграция                                  │
└──────────────────────────────────────────────────────────────┘
```

## Workspaces

- `apps/desktop` — Electron main/preload/renderer;
- `packages/shared` — IPC DTO и общие типы;
- `test-data` — синтетические KDBX fixtures;
- `docs` — архитектура, безопасность и roadmap.

Контракт KDBX-совместимости и PassDeck-owned metadata описан в [`DATA_COMPATIBILITY_CONTRACT.md`](DATA_COMPATIBILITY_CONTRACT.md).

## Состояние базы

Каждая открытая вкладка соответствует main-process session:

```text
sessionId
path
name
Kdbx | undefined
locked
readOnly
dirty
ownsLock
fingerprint(mtime,size)
```

Мастер-пароль не хранится отдельной строкой после открытия. Библиотека KDBX содержит производный credential material внутри объекта базы. При блокировке ссылка на объект KDBX удаляется.

## Сохранение

1. проверка dirty/readOnly/locked;
2. проверка fingerprint;
3. backup исходного KDBX;
4. `db.save()`;
5. временный файл в директории базы;
6. fsync;
7. rename;
8. обновление fingerprint.

## Пользовательские поля

Обычные пользовательские поля входят в `DatabaseView`. Для защищённых полей renderer получает только имя, флаг защиты и признак наличия значения. Само значение остаётся в main-процессе и выдаётся только отдельным вызовом `database:reveal-custom-field` при явном показе или копировании.

При редактировании защищённого поля renderer отправляет `preserveValue` и исходное имя поля. Main-процесс переносит существующее значение внутри объекта KDBX, поэтому секрет не требуется предварительно раскрывать в интерфейсе.

## Вложения

Renderer получает только имя и размер вложения. Выбор исходных файлов и пути экспорта выполняется системными диалогами в main-процессе. Байты вложения читаются, добавляются в `KdbxBinaries` и записываются на диск только внутри `DatabaseService`; через IPC содержимое файла не передаётся.

Для одной записи действует лимит 25 МБ на файл и 100 МБ суммарно. Совпадающие имена не перезаписываются автоматически. Добавление и удаление помечают базу изменённой; внутренние снимки KDBX не выводятся в отдельный пользовательский интерфейс.

## Перемещение и удаление

Renderer позволяет перетащить запись из центрального списка на другую группу в дереве. После drop отправляется типизированная IPC-команда с UUID записи и группы назначения. Main-процесс повторно проверяет существование обоих объектов и выполняет перемещение только внутри текущей открытой базы. Отдельных кнопок или модальных окон перемещения нет; перемещение групп не поддерживается.

Встроенная служебная группа KDBX Recycle Bin не включается в `DatabaseView` и не доступна как назначение. Удаление записи после подтверждения удаляет её из пользовательского дерева напрямую; отдельного интерфейса корзины и восстановления нет.

## Portable data root

На Windows electron-builder задаёт `PORTABLE_EXECUTABLE_DIR`. Поэтому settings root вычисляется как `<portable-dir>/data`.

В dev-режиме используется `data-dev`. В обычной непереносимой macOS-сборке используется `app.getPath('userData')/data`.

## Расширение архитектуры

Тяжёлый Argon2/KDBX поток должен быть вынесен в worker thread или Electron utility process до поддержки больших баз. IPC API при этом не меняется: renderer получает progress events и итоговый DTO.

KDBX load/create/save уже проходят через `KdbxOperations`, чтобы будущий worker-boundary не менял `DatabaseService` и renderer IPC. Текущую baseline-производительность можно снять командой:

```powershell
npm run benchmark:kdbx
```
