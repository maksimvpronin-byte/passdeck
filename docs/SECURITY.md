# Модель безопасности PassDeck

## Защищаемые активы

- мастер-пароль;
- производные ключи;
- пароли и защищённые поля записей;
- вложения;
- целостность KDBX;
- резервные и recovery-копии.

## Основные угрозы

- вредоносный KDBX/XML;
- XSS в renderer;
- произвольный IPC;
- загрузка удалённого контента;
- подмена файла другим приложением;
- одновременная запись двумя процессами;
- утечка через буфер обмена;
- утечка в настройки или логи;
- повреждение файла при аварии;
- supply-chain уязвимости.

## Текущие меры

- `contextIsolation`, sandbox, запрет Node.js в renderer;
- CSP и запрет навигации;
- белый список IPC;
- KDBX и криптография в main;
- Argon2id через WebAssembly;
- `npm audit` в build scripts;
- override уязвимой транзитивной XML-зависимости на исправленную версию;
- атомарное сохранение;
- две резервные копии;
- fingerprint перед сохранением;
- lock-файл;
- очистка буфера только если значение не менялось;
- отсутствие файловых логов;
- отсутствие сетевых запросов приложения.

## Известные ограничения каркаса

- расшифрованный KDBX живёт в памяти main-процесса до блокировки;
- JavaScript не гарантирует немедленное zeroization памяти;
- тяжёлые KDF-операции пока выполняются в main event loop;
- Auto-Type и Touch ID ещё не реализованы;
- нет независимого security audit;
- compatibility suite с KeePassXC должен быть расширен.

## Правила для следующих изменений

- не добавлять generic `invoke(channel, payload)`;
- не передавать секреты через URL, query string или DOM attributes;
- не использовать `localStorage` для секретов;
- не включать devtools в production по умолчанию;
- не писать секретные поля в исключения;
- не добавлять analytics/crash reporters;
- проверять архивы и импортируемые файлы на размер до загрузки в память;
- для временных вложений использовать случайное имя, ограниченные права и гарантированное удаление.
